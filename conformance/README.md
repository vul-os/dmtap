# DMTAP Conformance

The conformance suite is the **operational definition** of "DMTAP-compatible": an
implementation conforms at a level (Core / Private / Groups&Files / Legacy / Clients / Auth,
see spec В§10) if and only if it passes the corresponding cases here. It also covers the optional,
capability-negotiated **DMTAP-PUB extension** (В§22, `pub-1`) and the **CAD/artifact profile**
built on top of it (В§23) вЂ” additive, capability-gated families layered on the Core level exactly
as the optional push-wake (`PUSH`) family is (В§10.2, В§21.22).

The suite has two coupled artifacts plus this guide:

| File | Role |
|------|------|
| [`SUITE.md`](SUITE.md) | The **normative test-case catalog**: every case numbered (`DMTAP-<CAT>-<NN>`), grouped by В§10.3 level, with its spec clause, input, expected result (accept / reject + В§21 error code), and MUST/SHOULD. |
| [`suite.json`](suite.json) | The **machine-readable** form of those cases (mirroring their case ids; see the sync note below) вЂ” a runner in any language drives it. |
| [`vectors/vectors.json`](vectors/vectors.json) | The **byte-exact known-answer inputs** the Core/Private/Groups&Files/Legacy/Clients/Auth vectored cases dispatch on. |
| [`vectors/pub_vectors.json`](vectors/pub_vectors.json) | The **byte-exact known-answer inputs** the `PUB` (В§22) vectored cases dispatch on вЂ” a **separate file**, generated by [`vectors/gen_pub_vectors.py`](vectors/gen_pub_vectors.py), not by `dmtap-core` (see [Provenance & drift protection](#provenance--drift-protection)). |

`SUITE.md` catalogs **157** numbered cases and `suite.json` mirrors all **157** вЂ” the two are **in
sync** (the wave-2 deniable-1:1 and KT-v1-hardening families, the `PROFILE` display-data cases, the
optional `PUSH` wake-signaling cases, the `FILE` durability cases, the wave-3 device-cluster
`SYNC`, `ALIAS`, and gateway-alias `GWALIAS` families, the wave-4 DMTAP-PUB `PUB` family and the
CAD/artifact profile `CAD` family are all mirrored) and MUST stay so.
**52 cases are byte-runnable today** (46 backed by a `vectors.json` or `pub_vectors.json` entry, 6
self-contained CBOR-reject cases whose bytes are inline); 104 are `construction-todo` вЂ” each
carries an exact construction recipe and the expected В§21 error, and becomes byte-backed as its
subsystem gains a fixed-input KAT (see [Coverage vs. deferred](#coverage-vs-deferred)); 1
(`DMTAP-PUB-21`) is `manual-attestation` вЂ” a client-UX MUST with no wire bytes to recompute.

### How a third-party implementer uses the suite

1. Parse `suite.json`. For each case, read `level`, `operation`, `expect.outcome`
   (`match` / `accept` / `reject`), and вЂ” for `reject` вЂ” `expect.error_code`/`error_name`.
2. If the case has a `vector` (or `reuses_vector`), load that entry from `vectors.json` вЂ” or, for a
   `PUB`/`CAD` case, from [`vectors/pub_vectors.json`](vectors/pub_vectors.json) вЂ” and dispatch on
   its `operation` (see [How an implementation runs the vectors](#how-an-implementation-runs-the-vectors)
   and, for `PUB`, [How an implementation runs the PUB vectors](#how-an-implementation-runs-the-pub-vectors)).
   If it has an inline `input.cbor_hex`, feed those bytes to your deterministic-CBOR decoder.
   If it is `construction-todo`, build the fixture from the `construction` recipe. If it is
   `manual-attestation` (currently only `DMTAP-PUB-21`), there are no wire bytes to recompute вЂ”
   verify the client-UX requirement by review.
3. Assert the outcome: a `match` case recomputes to the committed value; an `accept` case validates;
   a `reject` case fails **and** maps to the named В§21 code with that code's disposition
   (DROP_SILENT / DEFER_REQUESTS / FAIL_CLOSED_BLOCK / HALT_ALERT / ACK_DEDUP / ROTATE_RETRY /
   DENY_POLICY).
4. You conform at a level iff every `MUST` case at that level (and every level it composes) passes.
   `pub-1` (the `PUB`/`CAD` families) is an **optional capability**, not a level: its guards are
   MUST *only* for an implementation that advertises `pub-1` (В§10.2, В§21.22, В§21.24b) вЂ” the
   identical treatment as the optional `push-wake` (`PUSH`) capability.

## Byte-exact known-answer vectors

[`vectors/vectors.json`](vectors/vectors.json) contains **byte-for-byte known-answer vectors**
for DMTAP's deterministic, security-critical operations. They make the spec
**machine-verifiable**: any independent implementation can compute the same operation over the
same fixed input and check that it gets the same committed answer.

The vectors are **not hand-written** вЂ” they are computed by the reference crate `dmtap-core`
(`../../envoir/crates/dmtap-core/`) and are proven correct against it by a self-checking test
(see [Provenance & drift protection](#provenance--drift-protection)). Once published, **the
vectors вЂ” not the reference вЂ” are normative** (spec В§10): a divergence between the reference and
these vectors is a bug to reconcile.

### Fixed-seed methodology

Every value is derived from a **fixed seed or fixed input** so it is exactly reproducible:

- **Ed25519** is deterministic per **RFC 8032** вЂ” a given secret seed + message always yields the
  same signature. Keys are reconstructed from fixed 32-byte seeds (`IdentityKey::from_seed`),
  never generated randomly.
- **BLAKE3-256** (content addresses, key-names, safety fingerprints) and **CBOR** (RFC 8949) are
  deterministic functions of their input.
- Two of the sign vectors reuse the **RFC 8032 В§7.1 Test 1 / Test 2** secret seeds, so an
  implementation gets a free cross-check against a published external standard (the committed
  public keys `d75a9801вЂ¦` / `3d4017c3вЂ¦` and signatures `e5564300вЂ¦` / `92a009a9вЂ¦` are exactly the
  RFC's values).

**Nothing that depends on fresh randomness is vectored as a full object.** HPKE encapsulation
uses an ephemeral key from the OS CSPRNG, so a sealed `Envelope.ciphertext` is not reproducible;
instead the *deterministic sub-operations* it decomposes into вЂ” the content-address check, the
`sender_sig` preimage/signature, and the `payload_sig` preimage/signature вЂ” are each vectored as
known-answer tests.

### Vector file format

`vectors.json` is a single object: `{ format, suite, generated_by, methodology, vectors: [...] }`.
Each entry in `vectors` is:

```json
{ "name": "вЂ¦", "operation": "вЂ¦", "input": { вЂ¦ }, "expected": { вЂ¦ }, "note": "вЂ¦" }
```

`name` is unique. `operation` names the algorithm a runner dispatches on. Byte strings are
lowercase hex. `input` fully determines the operation (for the input-driven operations); `expected`
is what a conforming implementation MUST produce.

## What the vectors prove

| # | Operation(s) | Vectors | What it proves |
|---|--------------|---------|----------------|
| 1 | `content_address` | `content_address_{empty,small,phrase,multi_kib}` | Content address = `0x1e вЂ– BLAKE3-256(bytes)` (spec В§2.2, В§18.1.5) for empty, small, phrase, and a 4096-byte input. |
| 2 | `content_address_verify` | `mote_content_address_{ok,tampered}` | The В§2.7 step-2 check: `id` recomputes from `ciphertext`, and a tampered ciphertext is rejected **before any decryption**. |
| 3 | `keyname_encode` / `keyname_verify` | `keyname_{zero_key,key_ones,key_twos,real_pubkey}`, `keyname_typo_rejected` | The zero-authority 8-word + checksum key-name (В§3.9.1, В§16.2): determinism, distinctness (`key_ones` в‰  `key_twos`), and that a mistyped word fails the folded checksum (fail-closed). |
| 4 | `safety_number` | `safety_number_pair_ab`, `safety_number_order_independent` | The В§3.4.1 OOB safety number: a deterministic fingerprint of a **pair** of identity keys, **order-independent** (swapping the two keys yields the identical value). |
| 5 | `ed25519_sign` / `ed25519_verify` | `ed25519_rfc8032_test{1,2}`, `ed25519_domain_separated`, `ed25519_verify_{ok,tampered_msg,tampered_sig}` | Deterministic Ed25519 signatures (incl. two RFC 8032 cross-checks and DMTAP's domain-separated `sign_domain`), and that verification fails closed on a tampered message or signature. |
| 6 | `cbor_encode` | `cbor_{identity,device_cert,payload,envelope}` | Exact deterministic CBOR bytes of a signed `Identity` (+ its content address), a `DeviceCert`, a `Payload`, and an `Envelope`. The self-check additionally round-trips each (`decode(encode(x)) == x`, and re-encode is byte-identical). |
| 7 | `suite_decode` | `suite_{reject_0x00,reject_0x04,reject_0x05,reject_0xff,accept_0x01,accept_0x02,accept_0x03}` | Suite fail-closed (В§1.1, В§18.1.4): an unregistered suite byte MUST be rejected on decode; `0x01`/`0x02`/`0x03` are known (reserved) ids that decode. |
| 8 | `ed25519_sign` / `ed25519_verify` | `mote_sender_sig`, `mote_sender_sig_verify`, `mote_payload_sig` | The two MOTE signature KATs (В§2.7 steps 3 & 8): the `sender_sig` preimage `id вЂ– to вЂ– ts_be64 вЂ– kind вЂ– challenge` under the ephemeral key, and the `payload_sig` preimage `BLAKE3-256(CBOR(payload with sig cleared) вЂ– u8(kind) вЂ– u64be(ts) вЂ– CBOR(to))` under the IK вЂ” both with their exact domain-separation labels. |

The **MOTE validation ordering** (В§2.7) is expressed as KATs #2 (content-address, checked before
decryption) and #8 (the two signature checks), which is the deterministic core of the ordered
recipient validation.

## How an implementation runs the vectors

1. Parse `vectors/vectors.json`.
2. For each vector, dispatch on `operation`, feed it the hex/field values in `input`, and compare
   your result to `expected`:
   - `content_address` в†’ `0x1e вЂ– BLAKE3-256(bytes_hex)` == `id_hex`.
   - `content_address_verify` в†’ recompute the address of `ciphertext_hex`; it matches `id_hex` iff `valid`.
   - `keyname_encode` в†’ your key-name of `pubkey_hex` == `name` (and it checksum-verifies).
   - `keyname_verify` в†’ checksum-verifying `name` == `checksum_verifies`.
   - `safety_number` в†’ your safety number / fingerprint of `(ik_a_hex, ik_b_hex)` == `safety_number` / `fingerprint_hex`.
   - `ed25519_sign` в†’ `Ed25519(seed_hex).sign(domain_hex вЂ– msg_hex)` == `sig_hex`, and the public key == `pubkey_hex`.
   - `ed25519_verify` в†’ verifying `sig_hex` over `domain_hex вЂ– msg_hex` under `pubkey_hex` == `valid`.
   - `suite_decode` в†’ decoding the CBOR in `cbor_hex` as a suite id succeeds iff `accepted`.
   - `cbor_encode` в†’ decode `cbor_hex`; re-encoding MUST be byte-identical (deterministic CBOR).

The reference runner is the crate's own self-check test, which is exactly this loop plus a drift
guard (below); use it as the executable specification of the dispatch.

## Provenance & drift protection

The vectors are generated and continuously verified by `dmtap-core` вЂ” from the **Envoir** repo
checkout (`envoir/crates/dmtap-core`), not from this one; this repo only carries the resulting
`vectors.json` as a committed artifact:

- **Generate:** from the Envoir repo root, `cargo run -p dmtap-core --example gen_vectors`
  recomputes `vectors.json` from the reference crate.
- **Self-check:** from the Envoir repo root, `cargo test -p dmtap-core` (test file
  `tests/conformance_vectors.rs`) proves, on every test run, that
  1. every input-determined vector **re-derives from `dmtap-core`** to its committed `expected`
     value (the vectors are correct against the reference, not guessed), and
  2. the committed `vectors.json` **byte-for-byte matches** what the current reference generates вЂ”
     so if a primitive ever changes, the test fails until the vectors are regenerated (**drift is
     caught**),

  plus CBOR round-trips and the suite fail-closed property directly.

The generator and the checker share one source of truth (`crates/dmtap-core/vectors_gen.rs.inc` in
Envoir), so they cannot silently disagree. Regenerating updates Envoir's copy; keeping this repo's
`conformance/vectors/vectors.json` in sync is a manual step when the reference changes.

## Coverage vs. deferred

**Covered (deterministic, known-answer):** content addressing; the key-name (encode + checksum);
safety numbers; Ed25519 sign/verify (incl. RFC 8032 cross-checks + domain separation); canonical
CBOR of `Identity` / `DeviceCert` / `Payload` / `Envelope` (+ round-trip); suite fail-closed; and
the MOTE content-address and signature KATs. **33 vectors** across 9 operations.

**Deferred, and why:**

- **HPKE seal/open (suite 0x01) and any full sealed `Envelope.ciphertext`** вЂ” RFC 9180 HPKE uses
  an ephemeral encapsulation key from the CSPRNG, so the output is non-deterministic. `dmtap-core`
  exposes no fixed-randomness HPKE path, so a byte-exact seal vector is not reproducible. Its
  deterministic decomposition (content-address + `sender_sig` + `payload_sig`) is covered instead.
  A fixed-KAT HPKE vector could be added if the reference gains a deterministic-RNG test seam.
- **MLS group operations (`GroupState`, `GroupEvent`, Welcome/Commit, В§5)** вЂ” MLS handshake bytes
  (RFC 9420) are carried opaquely by DMTAP and are not implemented in `dmtap-core`.
- **Suite `0x02` (PQ: ML-DSA-65 / X-Wing)** вЂ” reserved and unimplemented; it correctly fails
  closed, which *is* vectored (`suite_reject_*` covers unknown-suite rejection; a positive
  `0x02` object KAT awaits an implementation).
- **Argon2id PoW, ARC tokens, postage, vouches (В§9), key-transparency proofs (В§3.5), Merkle-DAG
  manifests (В§5.5), DMTAP-Auth challenge/response (В§13), nameв†’key DNS/KT resolution (В§3.3)** вЂ”
  either not yet implemented in the reference or dependent on external/interactive state; to be
  added as those subsystems land.

### A note on the CBOR encoding vectored here

The `cbor_encode` vectors are the **В§18-canonical, integer-keyed (COSE/CWT-style) deterministic
encoding** (RFC 8949 В§4.2; spec В§18.1.1 / В§18.1.2 / В§18.3.x / В§18.4.x), matching the
`vectors.json` `methodology` field and each vector's own note. This is directly visible in the
bytes: e.g. `cbor_identity` begins `a9 01 81 01 02 a1 вЂ¦` вЂ” a 9-entry map keyed by the small
unsigned integers `1, 2, 3, вЂ¦` in ascending order, exactly as В§18.4.1 specifies, **not** a
text-keyed map. A second implementer following В§18 alone reproduces these bytes; the self-check
additionally round-trips each object (`decode(encode(x)) == x`, byte-identical) and recomputes the
`Identity` content address (В§18.9.4). Cases `DMTAP-CBOR-01вЂ¦04` in `SUITE.md` pin these four
objects; `DMTAP-CBOR-05вЂ¦12` pin the canonical-form **reject** rules (non-preferred integers,
indefinite lengths, unsorted/duplicate keys, floats, `undefined`, null-valued optionals, and
unknown `в‰Ґ 64` keys in signed objects).

## The DMTAP-PUB (В§22) vectors вЂ” `pub_vectors.json`

[`vectors/pub_vectors.json`](vectors/pub_vectors.json) is the byte-exact known-answer companion to
the `PUB` cases (`DMTAP-PUB-01`вЂ¦`21`) and, indirectly (no bytes of its own, В§23.1), the `CAD`
checklist (`DMTAP-CAD-01`вЂ¦`11`). It is a **separate file** from `vectors.json` with its own
`format`/`suite`/`generated_by`/`methodology` header, kept distinct rather than merged into
`vectors.json` because the two have genuinely different provenance (below) вЂ” merging them under
one `generated_by` would misrepresent where the bytes came from.

### Provenance: a script, not `dmtap-core`

`vectors.json` is generated and drift-checked by the `dmtap-core` reference crate (above).
`pub_vectors.json` is **not** вЂ” it was written when `dmtap-core` did not yet implement the DMTAP-PUB
extension, and its bytes are still produced by the script, never by the crate. **That independence is
now a feature rather than a gap:** `dmtap-core` *does* implement DMTAP-PUB today (`dmtap_core::pubobj`,
В§22), and the Envoir conformance runner executes **every** vector in this file against it as a
mandatory gate вЂ” so the script and the crate are two independent implementations that agree, which is
exactly the cross-check a known-answer suite is for. Every operation
DMTAP-PUB needs for a byte-exact vector is fully deterministic вЂ” BLAKE3-256 content
addressing (the same primitive `vectors.json` uses) and Ed25519 signing over a domain-separated
preimage (the same `sign_domain` pattern as `mote_sender_sig`/`ed25519_domain_separated`) вЂ” so
[`vectors/gen_pub_vectors.py`](vectors/gen_pub_vectors.py), a ~300-line throwaway Python script
with **fixed** Ed25519 seeds (`0xAA`Г—32, `0xBB`Г—32), a **fixed** timestamp
(`1700000050000` ms, in the style of `vectors.json`'s own `mote_payload_sig` vector), and
**fixed** plaintext chunk bytes (no randomness, no wall-clock reads), reproduces every vector
below independently of any reference implementation. Regenerate/verify with:

```
pip install blake3 cryptography
python3 conformance/vectors/gen_pub_vectors.py > conformance/vectors/pub_vectors.json
```

Every value in this file was independently re-derived вЂ” a second, from-scratch implementation of
the same BLAKE3/Ed25519/CBOR-decode formulas that deliberately does not import
`gen_pub_vectors.py` вЂ” by [`vectors/verify_pub_vectors.py`](vectors/verify_pub_vectors.py) during
the wave-4 review that added it (`python3 conformance/vectors/verify_pub_vectors.py`; exits
nonzero on any mismatch). A divergence between the generator, the verifier, and the committed file
is a bug to reconcile, the same rule В§10 states for `vectors.json`/`dmtap-core`.

### How an implementation runs the PUB vectors

1. Parse `vectors/pub_vectors.json` (a separate parse from `vectors/vectors.json`).
2. For each vector, dispatch on `operation`:
   - `pub_manifest_root` в†’ hash each `plaintext_chunks_hex[i]` to `h_i = 0x1eвЂ–BLAKE3-256(...)`
     (== `chunk_hashes_hex[i]`), fold them into the В§22.2.2 DS-tagged tree, and check the root ==
     `id_hex`.
   - `pub_manifest_type_mismatch` в†’ root the given `chunk_hashes_hex` both the public (DS-tagged)
     and the В§18.9.5 bare-sealed way; assert `public_root_hex` в‰  `sealed_style_root_hex`.
   - `det_cbor_decode_pub_manifest` / `det_cbor_decode_feed_entry` в†’ your deterministic-CBOR
     decoder of a `PubManifest`/`FeedEntry` MUST reject `input.cbor_hex`.
   - `ed25519_sign` в†’ identical dispatch to `vectors.json`'s `ed25519_sign`:
     `Ed25519(seed_hex).sign(domain_hex вЂ– msg_hex)` == `sig_hex`, public key == `pubkey_hex`.
   - `content_address` в†’ identical dispatch to `vectors.json`'s `content_address`:
     `0x1eвЂ–BLAKE3-256(bytes_hex)` == `id_hex`.
   - `pub_supersede_check` / `pub_feed_anti_rollback` в†’ validation predicates; compare your
     accept/reject decision (and, on reject, `error_code`/`error_name`/`action`) to `expected`.
   - `pub_feed_entry_root` в†’ recompute each `entries_cbor_hex[i]`'s content address; check against
     `entry_ids_hex[i]` and that the `prev` fields resolve correctly along the chain.

There is no `dmtap-core`-equivalent self-checking test for these vectors in this repo (no Rust
reference implements DMTAP-PUB yet); `gen_pub_vectors.py` **is** both the generator and, by
re-running it and diffing, the drift check.

## The substrate Sync capability vectors вЂ” `sync_vectors.json` (informative companion)

[`../substrate/SYNC.md`](../substrate/SYNC.md) is an **additive substrate companion document**, not a
numbered section (`substrate/README.md`); its own В§10 conformance-vector table is the authoritative
index. [`vectors/sync_vectors.json`](vectors/sync_vectors.json), generated by
[`vectors/gen_sync_vectors.py`](vectors/gen_sync_vectors.py), freezes **all 24** of its vectors вЂ” the same
throwaway-script provenance model as `pub_vectors.json` above (the bytes come from the script, not
from a reference crate; an independent Rust implementation of `SYNC.md` now exists and executes them,
and it is what surfaced the `SYNC.md` В§14 C-01вЂ¦C-07 corrections; C-08вЂ¦C-10 came instead from the first real product adoption of the engine). The five that were previously NOT-FROZEN вЂ”
`SYNC-OP-02` (the `COSE_Sign1` envelope), `SYNC-TREE-01` (which side of a concurrent-move cycle loses),
`SYNC-SNAP-01`/`SYNC-SNAP-02` (the canonical "observable state" schema) and `SYNC-RECON-01` (the
range-Merkle fingerprint fold) вЂ” were resolved by adding normative text to `SYNC.md` **first** (В§4.1,
В§4.8, В§6.1.1, В§5.3, each with its rationale) and only then vectored, so no decision in the vector file
originates in the vector file. `SYNC-FJ-01`/`SYNC-FJ-02` (the В§5.2.1 fast-join `pull` response) are the
two newest, added with correction **C-05**: the same implementation found that `SYNC.md` В§5.2 had no way
to tell a peer "you are below my В§6.2 truncation floor, fast-join from this snapshot", so a truncating
responder could only return a suffix that silently lost the truncated ops while appearing to succeed.
В§5.2.1 decided the response shape and the MUST NOT first; the vectors followed. Corrections **C-06** and
**C-07** then tightened `SYNC-FJ-02` without changing the count (still 22): C-06 pins op framing inside an
`ops` array to **item-embedded `COSE_Sign1`**, never `bstr`-wrapped (`SYNC.md` В§5.2), and the vector's ops
are now **real signed `COSE_Sign1` envelopes** rather than bare `SyncOp` stand-ins вЂ” a stand-in that lacks
the specified shape is how the ambiguity survived a frozen vector in the first place; C-07 removes the
unverifiable "`floor` MUST NOT exceed `covers`" clause (a single `Hlc` and a per-author `VersionVector`
are not comparable) in favour of the new В§5.2.2 split between what a caller verifies and what it trusts.
`SYNC-SNAP-03` and `SYNC-VAL-01` take the count to **24**, and they came from a different place than
every correction before them: the **first real product adoption** of the Sync engine (ofisi's Sheets
grid), rather than an independent reimplementation of the document. **C-09** found that the snapshot
*body* is a compacted **op set**, not `det_cbor(ObservableState)` вЂ” В§6.1.1's projection deliberately
drops the merge metadata the next merge needs (the LWW winner's HLC above all), so a replica that
adopts it as base state diverges silently; `SYNC-SNAP-03` freezes the exact ordering case, with a
post-`covers` op that is genuinely after `covers` yet *below* the incumbent in the В§3 total order.
**C-08** found that В§4.1 named В§18.3.6's `ext-value` but described a strictly narrower type, blocking
nested application data twice over for no determinism reason; `SYNC-VAL-01` freezes the boundary from
both sides, including a depth-2 reject that pins validation as recursive. **C-10** (В§4.10's
death-certificate-vs-LWW selection guidance) adds no vector вЂ” `SYNC-DEATH-01` already pins the
domination it warns about. None of the previously frozen bytes changed.

**C-11вЂ¦C-14** keep the count at **24** and come from the implementation of C-08/C-09. **C-11** is a
regeneration rather than a new vector: C-09 recorded that making the body an op set left `SYNC-FJ-01`
alone "because key `3` is absent", which is true of that vector's `pull_response_cbor_hex` and false
of the `вЂ¦_with_inline_state_вЂ¦` variant it also carries вЂ” so a frozen vector was pinning, as the
inline body, the one value В§6.1.2 says a caller must not adopt. Key `3` now carries a real
`SnapshotBody` folding to the unchanged `root`, and the old framing is kept as a labelled
non-conformant artifact. **C-12**/**C-13(a)** widen `SYNC.md` В§6.2's retention set (a winning `Live`
death cell; a retained tombstoned RGA origin's own `seq-remove`; **every** PN-counter op, since no
counter delta is ever superseded) and are vectored only indirectly, by `SYNC-SNAP-03`'s
fold-then-recompute вЂ” the check that catches a body missing any of them. **C-13(b)** adds the
`sync-1/ext-value-2` sub-token C-08's mixed-deployment warning lacked, specified as **advisory**:
absence means "unknown", never "old", and no node may refuse or downgrade on it. **C-14** extends
`SYNC-VAL-01` with the empty map `0xa0` (key-type-ambiguous but vacuously so, and legal, so it MUST
be accepted) and the empty array, and fixes the nesting-depth ceiling at **64** as a MUST covering
all sync decoding. Again no previously frozen byte changed, apart from the C-11 artifact that was
the defect.
